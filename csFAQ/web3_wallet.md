## 2. 区块链与分布式账本基础

### 2.1 什么是区块链？
区块链是一个分布式账本数据库，它通过密码学手段将数据块按时间顺序相连，形成链式结构，并保证不可篡改和不可伪造。其三大核心支柱为：
1. **分布式对等网络（P2P）**：数据存在于全球成千上万个节点中，无单点故障，无中心化控制机构。
2. **共识机制（Consensus）**：网络参与者通过特定算法对账本状态达成一致。
   - **工作量证明（PoW）**：通过计算哈希碰撞竞争记账权，安全性极高但消耗能源。
   - **权益证明（PoS）**：通过质押代币（Staked Tokens）选择验证者（Validators）来生产区块，如以太坊 The Merge 之后的机制。
3. **密码学哈希与链条**：每个区块包含前一个区块的哈希值。若修改历史数据，会导致其后所有区块哈希失效，从而实现防篡改性。

### 2.2 RPC 节点：开发者的网关
对于软件工程师而言，**RPC（Remote Procedure Call）节点**是与区块链交互的 API 终结点。它充当了应用和去中心化账本之间的接口。

RPC 节点提供两种主流通信模式：
- **HTTP (RPC Calls)**：同步的“请求-响应”模式，类似于 REST API。用于查询当前状态、获取交易回执或发送交易。
- **WebSocket (WS Subscriptions)**：基于事件驱动的双向通信。用于订阅链上新区块、特定地址的余额变动或交易状态事件。

#### RPC 节点的核心职责
- **状态查询**：检查钱包余额、读取智能合约状态。
- **历史检索**：获取某一区块内的所有交易，或者根据交易哈希查询细节。
- **交易广播**：将本地签名好的交易（Raw Transaction）广播给矿工/验证者的 mempool（内存池）。
- **节点运行商**：如 Infura, Alchemy, QuickNode, 或自建全节点。

## 3. 钱包范式分类：EOA 与 AA

根据私钥控制权的归属以及账本账户类型的不同，钱包主要分为以下几大类：

### 3.1 托管与非托管（Custodial vs Non-custodial）
- **托管钱包（Custodial）**：私钥由第三方（如币安、OKX 交易所）保管。用户通过用户名密码登录，安全性完全依赖于托管方。
- **非托管钱包（Non-custodial）**：私钥由用户自己完全控制。若私钥丢失，资产无法找回。

### 3.2 外部账户（EOA）与合约账户（AA）
在以太坊（EVM）体系中，存在两种账户类型：
1. **EOA（Externally Owned Account，外部拥有账户）**：
   - 由一对非对称密钥直接控制。
   - 在链上没有代码，其状态仅由 `(balance, nonce)` 构成。
   - 必须作为交易的**发起者**（Transaction Initiator），且必须持有原生代币（如 ETH）以支付 Gas。
2. **AA/CA（Account Abstraction / Contract Account，智能合约账户）**：
   - 账户本身是一个部署在链上的智能合约，地址是合约地址。
   - 账户逻辑由合约代码决定（可编程验证、多签、限额等）。
   - 自身不能直接发起交易（必须由 EOA 触发，或者在 ERC-4337 下由 Bundler 代为发起），支持第三方代付 Gas。

### 3.3 EOA vs AA 全方位技术对比

| 维度 | EOA 钱包 | AA 智能合约钱包 |
| :--- | :--- | :--- |
| **物理实体** | 一对椭圆曲线密钥对 | 部署在链上的一段合约代码 |
| **地址来源** | 公钥哈希派生（$Address = \text{Keccak256}(P)[12..32]$） | 合约部署地址（通过 `CREATE` 或 `CREATE2`） |
| **签名验证** | 协议硬编码，强制 ECDSA（secp256k1） | 自定义，可由合约逻辑指定（如 P-256、多签、Passkey） |
| **Gas 支付** | 必须由该账户持有原生代币（ETH/X-Layer OKB） | 可由第三方代付（Paymaster），或用任意代币（如 USDC）抵扣 |
| **多笔操作** | 必须分步发送，每步签名并串行上链（Approve + Swap） | 支持多笔调用打包在单个 UserOperation 中一次性执行 |
| **私钥丢失** | 永久失去账户控制权，资产归零 | 支持社交恢复、紧急监护人变更、多签更新控制权 |
| **合约逻辑升级** | 无法升级（非代码） | 可以通过代理模式（如 Proxy/UUPS）安全升级 |
| **链上成本** | 极低（转账仅需固定 21,000 gas） | 较高（需支付合约创建、解析、验证等额外 gas） |
| **用户体验** | 助记词备份，理解成本高，体验偏 Web3 原生 | 无助记词登录（Passkey），Web2 风格登录 |

## 4. 交易、Gas 费用与 Nonce 机制

交易是钱包与区块链交互的唯一手段。对于软件工程师，交易是具有特定字段的结构化数据对象。

### 4.1 交易的生命周期

```
    [1. 交易创建] ───► [2. 签名] ───► [3. RPC 广播] ───► [4. 进入 Mempool]
                                                               │
                                                               ▼
    [7. 交易最终性] ◄── [6. 确认与追加区块] ◄── [5. 打包上链] ◄─── [矿工/验证者校验]
```

1. **创建 (Creation)**：客户端（如钱包 SDK）构造交易体（包含 `to`, `value`, `data`, `gasLimit`, `nonce` 等）。
2. **签名 (Signing)**：使用发送者的私钥对交易哈希进行 ECDSA 签名，生成带有签名 `(r, s, v)` 的原始交易数据。
3. **广播 (Broadcast)**：通过 RPC 节点的 `eth_sendRawTransaction` 将交易发送至网络。
4. **内存池 (Mempool)**：交易在被打包前进入分布式的优先级队列——Mempool 挂起等待。
5. **打包上链 (Inclusion)**：矿工或验证者根据 Gas Price 高低将交易从 Mempool 选出，打包进新区块。
6. **确认 (Confirmation)**：区块广播至全网，并在其后追加新的区块。确认数越多，重组风险越低。
7. **最终性 (Finality)**：在 PoS 中，当区块获得绝对多数验证者签名（如以太坊的 Epoch 确认）后，便具有不可逆转的最终性。

### 4.2 Nonce：串行化与防重放
在 EVM 中，每个账户在链上状态都存有一个 `nonce` 值，代表**该账户已成功发送的交易总数**（从 0 开始计数）。

#### 节点执行的验证逻辑
当节点收到一笔交易时：
1. 通过 `ecrecover` 恢复发送者地址 $A$。
2. 读取链上状态 $A$ 的当前 $\text{nonce}_{\text{chain}}$。
3. 对比交易中的 $\text{nonce}_{\text{tx}}$：
   - $\text{nonce}_{\text{tx}} == \text{nonce}_{\text{chain}}$：接受，执行，随后 $\text{nonce}_{\text{chain}} \leftarrow \text{nonce}_{\text{chain}} + 1$。
   - $\text{nonce}_{\text{tx}} < \text{nonce}_{\text{chain}}$：直接拒绝，抛出错误 `"nonce too low"`。这是最直接的**防重放攻击**手段。
   - $\text{nonce}_{\text{tx}} > \text{nonce}_{\text{chain}}$：将交易挂起（Pending）并存入 Mempool 队列，直到缺失的中间 Nonce 交易被补齐后才会串行执行。

#### 业务层签名中的 Nonce 选型
在智能合约或链下业务授权签名中，也需要 Nonce 防止操作重放。有两种设计：

| 模式 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **递增式（Sequential）** | 逻辑极简，链上仅需存一个 `uint256` 计数器，节省 gas | 限制了并发性能。前面的签名未上链时，后续签名无法提交 | 权限变更、多签管理等必须保证严格顺序的场景 |
| **随机散列式（Random 32B）** | 允许高度的并发。只要 Nonce 不同，多笔交易可并行上链 | 每一个用过的 Nonce 都要在合约里占一个 `mapping(uint256 => bool)` 插槽，状态成本高 | 高频支付、批量转账授权（如 OKX Pay 订单） |

## 5. 密钥管理与 HD 分层确定性钱包

非对称加密中，公私钥是一对数学上关联的字节数组。由于私钥长达 32 字节且无规律，对人类用户极度不友好。为了便于存储和恢复，区块链行业制定了密钥派生标准。

```
              [ 128 / 256 位随机熵 ] ──( 加上校验和 )
                      │
                      ▼
               [ BIP-39 助记词 ] (12 / 24 个英文单词)
                      │
                      ▼ PBKDF2 ( HMAC-SHA512 + 密码 "mnemonic" )
               [ 512 位种子 (Seed) ]
                      │
                      ▼ BIP-32 算法 ( 派生主公私钥 )
                [ 主私钥 m ]
                      │
                      ▼ BIP-44 派生路径 ( m / purpose' / coin_type' / account' / change / address_index )
       ┌──────────────┴──────────────┐
       ▼                             ▼
  [ EOA 私钥 0 ]                [ EOA 私钥 1 ] (例如：m/44'/60'/0'/0/0 用于以太坊第一个地址)
```

### 5.1 BIP-39：助记词标准
BIP-39 将高强度的随机熵映射为 12 或 24 个可读单词。
- 首先生成 128 到 256 位的随机数，计算校验和并拼接。
- 将其每 11 位分割，对应到一个包含 2048 个标准英文单词的字典中。
- 利用 PBKDF2 算法（结合盐值 `"mnemonic" + 可选盐加密密码`），进行 2048 次 HMAC-SHA512 哈希，生成一个 64 字节（512 位）的 Seed。

### 5.2 BIP-32 / BIP-44：分层确定性钱包（HD Wallet）
- **BIP-32**：定义了如何使用单向 HMAC-SHA512 哈希，由 64 字节 Seed 递归派生出无数子密钥对（形成树状结构）。
- **BIP-44**：为派生路径规定了行业通用的语义层次：
  $$\text{m / purpose'} / \text{coin\_type'} / \text{account'} / \text{change} / \text{address\_index}$$
  - `purpose'`：固定为 `44'`。
  - `coin_type'`：币种标识。以太坊为 `60'`，比特币为 `0'`，Solana 为 `501'`。
  - `account'`：逻辑账户，便于区分同一用户的不同账户。
  - `change`：`0` 代表外部接收地址，`1` 代表找零地址。
  - `address_index`：具体的子地址索引。例如，以太坊第一个派生地址私钥位于 `m/44'/60'/0'/0/0`。

## 6. 非对称密码学与 ECDSA 签名

### 6.1 椭圆曲线与 secp256k1
以太坊和比特币核心采用 **secp256k1** 椭圆曲线。其代数方程定义为：
$$y^2 = x^3 + 7 \pmod p$$
其中 $p$ 为一个极大的特殊素数：
$$p = 2^{256} - 2^{32} - 977$$
曲线上定义了一个固定的基点（生成元） $G(G_x, G_y)$。

#### 单向乘法门（点乘）
私钥 $d$ 是一个落在 $[1, n-1]$ 区间内的 32 字节大整数（$n$ 为曲线的阶，接近 $2^{256}$）。通过标量乘法求得公钥 $P$：
$$P = d \times G$$
- **私钥到公钥**：极易。通过“双倍并相加（double-and-add）”算法，仅需约 256 次计算，毫秒级完成。
- **公钥到私钥**：极难。若已知公钥 $P$ 和基点 $G$，反推私钥 $d$ 属于**椭圆曲线离散对数问题（ECDLP）**，目前最好的经典算法复杂度约为 $2^{128}$，在经典计算机上是不可能解开的。

### 6.2 公钥格式与地址派生
公钥 $P$ 是一个二维坐标点 $(P_x, P_y)$，有压缩与非压缩两种表达：
- **非压缩格式（65 字节）**：$\text{0x04} \mid\mid P_x[32\text{ bytes}] \mid\mid P_y[32\text{ bytes}]$
- **压缩格式（33 字节）**：$\text{0x02} \text{ 或 } \text{0x03} \mid\mid P_x[32\text{ bytes}]$ （根据 $P_y$ 的奇偶性，通过曲线方程可在使用时解出 $P_y$）。

EVM 体系下，直接使用非压缩公钥去掉 `0x04` 前缀后的 64 字节数据进行哈希运算派生地址：
$$Address = \text{0x} \mid\mid \text{Keccak256}(P_x \mid\mid P_y)[12..32]$$
即取 Keccak-256 哈希结果的最后 20 字节。

### 6.3 ECDSA 签名与验证原理
对于消息哈希 $h$（32 字节）和私钥 $d$：
1. 选择一个临时的安全加密随机数 $k \in [1, n-1]$。
2. 计算曲线上的点 $R = k \times G = (x_R, y_R)$。
3. 计算签名分量 $r = x_R \pmod n$。
4. 计算签名分量 $s = k^{-1} \cdot (h + d \cdot r) \pmod n$。
5. 确定 recovery ID $v$（点的奇偶性，以及链 ID 信息）。
最终输出的签名字节串为 $(r, s, v)$，长度通常为 65 字节。

#### 密钥恢复（ecrecover）
ECDSA 具备一个独特性质：**从签名 $(r, s)$ 和消息哈希 $h$ 中，可以直接数学反推出签名者的公钥 $P$**：
$$P = r^{-1} \cdot (s \cdot R - h \cdot G)$$
EVM 提供了用于密钥恢复的内置预编译合约 `ecrecover`（位于地址 `0x0000000000000000000000000000000000000001`，调用消耗 \~3,000 gas）。它能直接解出公钥哈希派生的 20 字节地址，如果恢复失败则返回空地址 `0x0`。

```
                    ┌───────────────┐
                    │  ecrecover    │
 (签名, 消息哈希) ─►│ (预编译合约)  ├─► 恢复的以太坊地址 A
                    └───────┬───────┘
                            │
                      ( A == Expected_Signer ? ) ──► 验证成功 / 失败
```

#### ⚠️ 临时随机数 $k$ 的安全隐患
在 ECDSA 中，每次签名**必须**使用全新的随机数 $k$。如果同一个私钥 $d$ 对两个不同的消息 $h_1, h_2$ 签名时复用了同一个 $k$，攻击者可通过代数方法消去 $k$，直接推导泄露私钥 $d$：
$$s_1 - s_2 = k^{-1}(h_1 - h_2) \implies k = \frac{h_1 - h_2}{s_1 - s_2} \pmod n$$
$$d = \frac{s_1 \cdot k - h_1}{r} \pmod n$$
2010 年索尼 PlayStation 3 系统的签名机制就是因为将 $k$ 设为固定常数而被攻破。为此，目前的客户端大多遵守 **RFC 6979** 规范：即 $k$ 并非由硬件随机数生成器产生，而是通过私钥 $d$ 和消息哈希 $h$ 的 HMAC 确定性生成。只要私钥或消息变动，产生的 $k$ 就会发生剧烈随机漂移。

## 7. 账户抽象（AA）深度剖析

### 7.1 ERC-4337：无共识层变动的标准化栈

ERC-4337 在不需要修改以太坊共识层的前提下，通过引入一条平行的链下内存池（Alternative Mempool）和入口合约，为以太坊实现了完整的账户抽象。

```
  ┌────────────────────────────────────────────────────────┐
  │                   链下 (Alternative Mempool)            │
  │  用户 ──► 签署 UserOperation ──► 发送至 Bundler 节点    │
  └───────────────────────────┬────────────────────────────┘
                              │ 打包 handleOps()
                              ▼
  ┌────────────────────────────────────────────────────────┐
  │                   链上 (EntryPoint 合约)               │
  │                  0x0000000071727De...                  │
  │                           │                            │
  │             ┌─────────────┴─────────────┐              │
  │             ▼ 验证阶段                   ▼ 执行阶段     │
  │     SmartAccount.validateUserOp()      SmartAccount.   │
  │             │                          execute()       │
  │             ▼ (可选，由代付方代付)                      │
  │     Paymaster.validatePaymasterUserOp()                 │
  └────────────────────────────────────────────────────────┘
```

- **UserOperation**：用户操作描述符，是一个包含 `sender`、`nonce`、`initCode`、`callData`、`paymasterAndData`、`signature` 等字段的结构体。
- **Bundler（打包器）**：链下节点，收集备用内存池中的 UserOperations，并将它们打包为一笔标准的 EVM 交易，调用 EntryPoint 的 `handleOps` 函数。
- **EntryPoint（入口合约）**：经过严格审计并全局部署的单例合约，控制整个 AA 校验与执行生命周期。
- **Paymaster（付款人合约）**：允许灵活 Gas 支付策略的中间件。如果 UserOp 中指定了 Paymaster，EntryPoint 会向其查询是否愿意支付 Gas，从而实现 Gas 免除或以 ERC-20 代币扣款。

### 7.2 EIP-7702：EOA 的原地升级提案
为解决存量 EOA 无法直接享受 AA 合约便利性的问题，EIP-7702 提出在交易签名中引入特殊机制：
- EOA 账户签发一个授权，允许链上在其地址上临时附加一段智能合约字节码（类似代理合约）。
- 使得存量的 EOA 地址在执行该笔交易时具备可编程验证、批量处理等合约能力，无需用户进行繁琐的资产划转与地址迁移。

### 7.3 CREATE2 预测地址与惰性部署
智能账户的地址不由 `deployer` 的交易次数（nonce）决定，而是使用 `CREATE2` 指令计算得出：
$$\text{ContractAddress} = \text{Keccak256}(\text{0xff} \mid\mid \text{Deployer} \mid\mid \text{Salt} \mid\mid \text{Keccak256}(initcode))[12..32]$$
- **惰性部署（Lazy Deployment）**：由于地址是确定性的，用户可以在完全不花 Gas 部署合约的前提下，预先计算出自己未来的钱包地址。
- **流程**：用户充值 $\rightarrow$ 资金先到账 $\rightarrow$ 用户第一次调用钱包发送交易时，初始化代码（`initCode`）作为 UserOp 提交给 EntryPoint $\rightarrow$ Factory 伴随首笔交易自动部署智能账户并扣除 Gas，完成注册。

## 8. Passkey 与 WebAuthn 硬件密钥管理

在 AA 钱包体系中，最令人激动的实践是引入 **Passkey（WebAuthn 标准）** 彻底终结助记词。

### 8.1 secp256k1 vs P-256 (secp256r1)
传统的以太坊底层仅对 **secp256k1** 曲线提供原生预编译硬支持（`ecrecover`）。但是，所有智能手机的安全芯片（如 Apple 的 **Secure Enclave**、Android 的 **Keystore TEE** 以及计算机的 **TPM 芯片**）都只支持 NIST 推荐的 **P-256 (secp256r1)** 曲线，不支持 secp256k1。

#### 为什么只能选 P-256？
- **硬件限制**：硬件芯片在出厂时将 P-256 算法写死在了硅片中，私钥在安全芯片内部生成且永不出芯片。
- **验证成本**：
  - 如果链上直接用 Solidity 模拟 P-256 验证，每次验签需要消耗 $> 100,000$ gas，极其昂贵。
  - **RIP-7212 规范**：针对 P-256 签名推出了高效的预编译合约（X Layer 上部署仅消耗 \~3,450 gas），使手机生物验证（Face ID/指纹）直接作为链上 Owner 成为可能。

### 8.2 Passkey 注册与签名序列

```
【注册流程】
  浏览器/App                     操作系统 (SEP/TEE)               链上合约 (AA)
      │                               │                            │
      ├─► 1. 触发创建请求 (Challenge) ─►│                            │
      │                               ├─► 2. 硬件生成 P-256 密钥对    │
      │                               │   (私钥锁死在硬件中)        │
      │◄─ 3. 返回 credentialId, 公钥 ─┤                            │
      │                               │                            │
      └───────────────────────────────┼───────────────────────────►│ 4. 通过 Factory 写入
                                                                     (作为合约 Owner)

【签名验证流程】
  链上合约 (AA)                  浏览器/App                       操作系统 (SEP/TEE)
      │                               │                               │
      │◄── 5. 发起支付操作 ───────────┤                               │
      │                               ├─► 6. 请求生物识别 ───────────►│
      │                               │   (验证 Face ID / Touch ID)   │
      │                               │◄─ 7. 芯片内签名 (r, s) ───────┤
      │◄── 8. 提交 4337 交易 ─────────┤
      │    (携带 clientData / sig)
      │
      ├─► 9. 调用 RIP-7212 预编译 ──┐
      │   (校验 P-256 签名)         │
      │◄────────────────────────────┘
      │
      ▼
   [交易执行成功]
```

1. **凭证注册**：调用 `navigator.credentials.create`。手机拉起生物验证（Face ID/指纹），安全芯片生成 P-256 密钥对，私钥永远锁死在安全芯片中。公钥发送至链上，作为 AA 智能合约的 `Owner` 公钥存入状态中。
2. **签名授权**：调用 `navigator.credentials.get`。手机用生物验证确认用户在场（User Presence）后，将待签名的 Hash送入芯片，安全芯片输出 ECDSA $(r, s)$ 签名。
3. **安全保证**：由于 WebAuthn 强制校验 origin，天然杜绝了来自钓鱼网站（Phishing Domain）的签名劫持。

## 9. Spending Policy 与 2-2 双签安全网关

虽然智能合约可以通过 Passkey 赋予用户安全便利的控制权，但在高频自主交易或自动化场景中，用户不希望每次交易都手动拉起 Face ID。这就需要建立 **Owner 分权治理结构**。

### 9.1 Owner 角色分工

```
                 ┌─────────────────────────────────┐
                 │       AA 智能账户 (合约)         │
                 └────────────────┬────────────────┘
                                  │
         ┌────────────────────────┴────────────────────────┐
         ▼ 管理权 (高安全/低频)                              ▼ 支配权 (受限/高频)
   ┌───────────┐                                      ┌───────────┐
   │ 用户密钥  │ ─► Passkey (P-256)                   │ 代理密钥  │ ─► Agent (secp256k1)
   └───────────┘    - 修改策略                         └───────────┘    - 仅可花 token_whitelist
                    - 注册/撤销 AccessKey                              - 单笔限额 max_per_tx
                    - 转移全部资金                                     - 周期限制 window_seconds
```

- **用户管理私钥（Owner #0）**：通常为硬件 P-256 密钥（Passkey）。拥有账户最高权限，用于修改合约规则、注册或撤销受限的 Agent 密钥。
- **Agent 代理私钥（Owner #1）**：通常为 secp256k1 软件私钥，保存在 Agent 的本地 Keychain 中。可在受到 `SpendingPolicy` 严格规约的前提下快速扣款。

### 9.2 Spending Policy 数据结构
```solidity
struct SpendingPolicy {
    uint256 max_per_tx;        // 单笔最高限额
    uint256 max_total;         // 单个时间窗口内累计花费上限
    uint256 window_seconds;    // 时间窗口长度（如：30 天 / 86400 秒）
    uint256 spent;             // 在当前时间窗口内已消费的金额
    address[] token_whitelist; // 允许划扣的代币白名单地址列表
    uint64  expiry;            // 代理密钥的失效截止时间戳
}
```

### 9.3 双层安全 Gate 机制
在企业级网关（如 OKX Pay）中，支付安全通过“链下风控 + 链上硬性策略”双重 Gate 进行保障：

```
       [ 1. 发起划扣指令 ]
               │
               ▼
   ┌──────────────────────┐
   │  Gate ① 链下风控过滤  │ ──► 检查用户 IP、设备指纹、KYC 状态、AML 反洗钱风险
   └───────────┬──────────┘
               │ (后端批准签名)
               ▼
   ┌──────────────────────┐
   │  Gate ② 链上合约硬约束│ ──► 智能合约校验 SpendingPolicy：
   └───────────┬──────────┘     - 是否超额？代币是否在白名单？密钥是否过期？
               │
               ▼
     [ 3. 链上执行转账 ]
```

#### 2-2 双签机制
- 代理密钥（Agent Private Key）签名 + 后端风控协同服务（Collaborator/Server）签名。
- 两个签名同时提交给链上账户进行验证。即使 Agent 客户端私钥被盗，攻击者若无法绕过服务端的安全风控（未获得 Server 联签），交易依旧会在链上被 revert。同理，Server 单方面也无法动用用户资金，达成最小化信任协作。

## 10. 钱包领域核心 EIP 规范

在 Web3 交互开发中，为了保证钱包与 DApp 之间标准统一，必须遵循以下以太坊改进提案（EIPs）：

### 10.1 EIP-191：防伪造消息签名规范
防止用户被恶意 DApp 欺骗，误签署了一段伪装成“普通纯文本”的链上交易原始 Hex 数据。
EIP-191 强制在待签数据前插入特定的标志性前缀：
$$\text{SignedMessageHash} = \text{Keccak256}(\text{"\x19Ethereum Signed Message:\n"} \mid\mid \text{Length(Message)} \mid\mid \text{Message})$$
由于 `0x19` 字节在以太坊交易 RLP 编码格式的首字节是不合法的，因此任何 EIP-191 签名的消息哈希在链上都无法作为普通交易重放。

### 10.2 EIP-712：类型化结构体数据签名
EIP-191 仅能展示一段晦涩的十六进制文本。EIP-712 规范了**类型化数据签名**，使得钱包（如 MetaMask）可以为用户呈现人类可读的结构化内容：

```
+------------------------------------+
| 🔐 签名请求 (Sign Typed Data)       |
|                                    |
| 域分离 (Domain):                    |
| - 合约名: OKX Pay                   |
| - 链 ID: 196 (X Layer)             |
| - 验证合约: 0xabcd...1234          |
|                                    |
| 消息数据 (Message):                 |
| - 发送方: 0x1234...                |
| - 接收方: 0x5678...                |
| - 金额: 100.50 USDC                |
| - 订单 Nonce: 42                   |
| - 截止时间: 2026-06-13 18:30 UTC   |
+------------------------------------+
```

#### 实现原理解析
EIP-712 通过两个核心哈希拼接计算，彻底锁死签名域：
1. **Domain Separator（域分离哈希）**：
   将应用名、版本号、Chain ID 和合约地址哈希混合。保证该签名仅在该链、该特定合约内有效，杜绝跨链、跨应用重放。
   $$\text{domainSeparator} = \text{Keccak256}(\text{abi.encode}(\text{TypeHash}, \text{nameHash}, \text{versionHash}, \text{chainId}, \text{verifyingContract}))$$
2. **StructHash（结构体哈希）**：
   按预先定义的结构体字段类型进行打包哈希。
   $$\text{structHash} = \text{Keccak256}(\text{abi.encode}(\text{StructTypeHash}, \text{val}_1, \text{val}_2, \dots))$$
最终生成的哈希格式遵循 EIP-191 的 `0x1901` 子前缀：
$$\text{FinalHash} = \text{Keccak256}(\text{"\x19\x01"} \mid\mid \text{domainSeparator} \mid\mid \text{structHash})$$

### 10.3 EIP-3009：免 Gas 的授权划扣规范（USDC/USDT 标配）
为了消除用户钱包中必须备有 ETH 才能转账的痛点，EIP-3009 引入了链下签名授权机制（代币转账委派）。
1. 用户在链下，用 EOA 私钥签署一个类型化数据 `TransferWithAuthorization`（包含 `from`, `to`, `value`, `validAfter`, `validBefore`, `nonce`）。
2. 将此签名发送至 Relayer。
3. Relayer 支付 Gas 并调用代币合约的 `transferWithAuthorization` 函数。
4. 代币合约提取签名、校验是否被 `from` 签署，验证 Nonce 且时间未过期，随后在合约内部将余额划扣。用户体验为完美的 0 Gas 原生体验。

### 10.4 EIP-1271：智能合约账户的统一签名检验接口
外部 DApp（例如 Uniswap）通常仅接受 ECDSA 椭圆曲线签名（由 `ecrecover` 验证）。但合约钱包账户（AA）并没有对应的私钥，无法直接生成 ECDSA 签名。
EIP-1271 规范了统一的验证合约接口：
```solidity
interface IERC1271 {
    // 若签名有效，返回固定魔术值 0x1626ba7e = bytes4(keccak256("isValidSignature(bytes32,bytes)"))
    function isValidSignature(bytes32 hash, bytes calldata signature) external view returns (bytes4 magicValue);
}
```
当 DApp 检测到目标地址是一个 CA（合约账户）而非 EOA 时，会主动在链上调用该合约的 `isValidSignature`。合约可以在内部定制签名校验规则（例如：验证是否是其内嵌的 Passkey 公钥对该 hash 的签名）。

## 11. 工程案例：OKX Pay 链上架构与反推验证

通过分析 X Layer 主网上部署的 OKX Pay 真实钱包实例，可以完整反推出其工业级的架构选型：

```
                【 OKX Pay 链上合约拓扑 】
                
            ┌────────────────────────────────┐
            │   EIP-1967 Proxy (代理合约)     │
            │   (存储具体余额与个性化状态)    │
            └───────────────┬────────────────┘
                            │ 指向 (UUPS 模式)
                            ▼
            ┌────────────────────────────────┐
            │    Implementation (逻辑合约)    │
            └───────────────┬────────────────┘
                            │
            ┌───────────────┴───────────────┐
            ▼                               ▼
┌───────────────────────┐       ┌───────────────────────┐
│ EntryPoint (v0.7)     │       │ Signer Registry       │
│ 0x0000000071727De...  │       │ (独立账户管理合约)    │
└───────────────────────┘       └───────────────────────┘
  - 负责 4337 调度与执行          - 登记 K1 / R1 状态
  - 触发 validateUserOp           - 校验 isValidSignature (EIP-1271)
```

- **核心 AA 标准**：选用 **EIP-4337 v0.7** 规范，统一绑定全局部署的 EntryPoint 地址（`0x0000000071727De22E5E9d8BAF0edAc6f37da032`）。
- **代理架构**：使用 **EIP-1967 标准代理** 降低部署 Gas。
- **升级模式**：采用 **UUPS**（Universal Upgradeable Proxy Standard）模式，升级函数写入实现合约（Implementation）中，以减小代理合约的大小。
- **多方验证**：与独立的 `Signer Registry` 进行交互以管理 Owner。支持 EIP-1271 合约签名，兼容 P-256 和 secp256k1 跨链调度。
